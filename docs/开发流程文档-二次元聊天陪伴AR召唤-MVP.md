# 开发流程文档：二次元聊天陪伴 + AR 召唤 App MVP

## 文档信息

- 文档版本：`v0.2`
- 文档状态：`Draft`
- 更新时间：`2026-05-07`
- 关联 PRD：`PRD-二次元空间陪伴App-MVP.md`
- 关联技术设计：`技术设计文档-二次元聊天陪伴AR召唤-MVP.md`
- 产品阶段：`MVP`
- 客户端平台：`Android 原生`

## 1. 目标

本文档用于指导 MVP 从当前 PRD 与技术设计进入实际开发，目标是把产品拆成可执行阶段，明确每个阶段的输入、产出、验收标准和阻塞条件。

开发主线为：

`聊天陪伴主闭环 -> 角色资产接入 -> AR Optional 召唤 -> fallback 降级 -> 截图保存 -> 数据治理与安全策略 -> Alpha 测试`

## 2. 开发原则

- 聊天是主闭环，AR 是增强模块，不能为了 AR 推迟聊天 MVP。
- 先做可跑通的端到端闭环，再打磨视觉、动画和性能。
- AR 必须 Optional，不支持 AR 的设备不能被挡在产品外。
- 首版只做截图，不做录像、不做作品库、不做房间理解。
- 所有安全、隐私、权限、删除逻辑必须在 Alpha 前闭环。
- 每一阶段结束必须有可演示版本，不能只完成底层代码。

## 3. 阶段 0：工程基线整理

目标：让 Android 工程具备正式开发基础。

### 3.1 主要任务

- 统一包名、应用名、namespace、applicationId。
- 建立基础目录结构：`auth`、`chat`、`companion`、`summon`、`settings`、`core`。
- 接入 Compose Navigation。
- 接入基础 ViewModel、UI State、Repository 分层。
- 准备 Debug mock 数据开关。
- 确认 `minSdk=24`、`targetSdk`、权限声明策略。
- 建立基础 CI 编译检查，至少覆盖 Debug 包构建。
- 接入基础 lint 规则，阶段 0 要求新增代码无高优先级 lint 违规。

### 3.2 阶段产出

- App 可正常启动。
- 有基础页面导航框架。
- 有 mock 登录态与 mock 角色数据。
- 不依赖真实后端也能进入主页面。
- CI 能完成一次干净的 Debug 构建检查。

### 3.3 验收标准

- 冷启动无崩溃。
- 可从登录页进入聊天首页。
- 工程结构能支撑后续模块开发。
- Debug 构建检查通过，新增代码无阻断级 lint 问题。

## 4. 阶段 1：聊天陪伴主闭环

目标：先做出产品的核心日常使用体验。

### 4.1 主要任务

- 实现邮箱验证码登录 UI 与 mock 登录流程。
- 登录页必须包含隐私协议与数据处理说明的确认交互，未确认前不能进入登录提交。
- 实现角色创建页：昵称、语气、人设标签、兴趣标签。
- 实现聊天首页：消息流、输入框、发送状态、角色回复。
- 实现 mock LLM 服务，后续可替换真实接口；mock 至少支持正常模式与安全/受限模式两类 preset。
- 实现最近一次召唤/作品上下文在聊天中的展示入口。
- 实现基础记忆结构：关系标签、最近互动、最近作品主题。
- 实现安全策略占位：未成年人自述、危机内容、安全模式。

### 4.2 阶段产出

- 用户可以完成登录、创角、进入聊天、发送消息、收到角色回复。
- 聊天首条回复目标时间按 PRD 口径控制在 `<= 4 秒`。
- 聊天首页成为默认首页。

### 4.3 验收标准

- 首次聊天链路完整。
- mock LLM 下消息状态正确：发送中、成功、失败、重试。
- mock LLM 能通过测试输入切换正常回复与安全/受限回复。
- 未成年人或危机内容触发后，回复风格能切换到受限模式。
- 未确认隐私协议与数据处理说明时，登录提交入口不可用。

## 5. 阶段 2：角色资产与召唤入口

目标：在不依赖完整 AR 的情况下，先完成角色资产展示与召唤产品入口。

### 5.1 主要任务

- 确定首版唯一 3D 角色资产包。
- 首版 3D 角色资产输入源固定为 `E:\aic\feiyingfbx\未命名.fbx`。
- 完成 `FBX -> GLB` 离线转换链路。
- 输出 `character.glb` 与 `CharacterAssetManifest`。
- 接入基础 3D 渲染能力。
- 在聊天首页增加“召唤”入口。
- 增加官方锚点图查看/保存入口。
- 将项目内资源 `pictures\ar_marker_fantasy.png.jpeg` 作为官方锚点图 v1 候选资源，当前本机绝对路径为 `E:\projects\THETWO\pictures\ar_marker_fantasy.png.jpeg`。
- 在进入 AR 联调前确认锚点图实际格式、分辨率和文件命名，必要时统一重命名为明确格式的资源名。

### 5.2 阶段产出

- 角色模型能在测试页面稳定显示。
- `E:\aic\feiyingfbx\未命名.fbx` 已完成离线转换，客户端使用转换后的 `GLB` 产物。
- 至少有一个待机动画或静态待机场景。
- 用户能从聊天首页进入召唤页。
- App 内能查看官方锚点图说明。

### 5.3 验收标准

- 角色首帧出现时间满足 PRD 技术阈值。
- 模型加载失败有明确错误页和重试。
- 官方锚点图通过打印和第二屏显示的基础识别验证后，才能进入 AR 联调。
- 锚点图资源必须能从项目仓库路径访问，不能依赖用户个人下载目录。

## 6. 阶段 3：AR Optional 召唤主路径

目标：在支持设备上完成图像锚定 AR 召唤体验。

### 6.1 主要任务

- 接入 ARCore。
- 实现 AR Optional 检测流程。
- 支持 Google Play Services for AR 缺失、过旧、安装取消、安装失败处理。
- 接入 Augmented Images 图像锚定。
- 识别官方锚点图后自动显示角色。
- 支持相对位移、旋转、缩放、重置。
- 实现短时丢锚保持，超过阈值提示重新对准。
- 补齐 ARCore 隐私告知弹窗与设置页查看入口。

### 6.2 阶段产出

- 支持 AR 的设备可以完成：进入召唤页、识别锚点图、角色出现、调整角色。
- 用户取消安装或设备不支持时，不崩溃、不阻断，进入 fallback。

### 6.3 验收标准

- 图像锚定识别成功耗时满足 PRD 阈值。
- 角色放置完成率可埋点统计。
- AR 页面 crash-free rate 满足 PRD 阈值。
- 权限拒绝、相机占用、服务安装失败都有恢复路径。

## 7. 阶段 4：Fallback 降级召唤

目标：保证不支持真 AR 的 Android 设备也能完成召唤和截图闭环。

### 7.1 主要任务

- 实现 CameraX 相机预览叠加模式。
- 实现无相机权限时的纯屏幕模式。
- fallback 中支持角色拖动、缩放、旋转、重置。
- fallback 页面复用召唤页主要 UI。
- 区分 AR supported funnel 与 fallback funnel 埋点。

### 7.2 阶段产出

- 非 AR 设备仍可完成角色摆放与截图。
- 用户不会因为设备不支持 AR 而无法体验核心功能。

### 7.3 验收标准

- 权限拒绝后仍可进入纯屏幕模式。
- fallback 截图可保存。
- fallback 漏斗数据与真 AR 漏斗分开统计。

## 8. 阶段 5：截图保存与聊天回流

目标：完成“召唤 -> 截图 -> 回到聊天 -> 角色引用刚才作品”的闭环。

### 8.1 主要任务

- 使用 MediaStore 保存截图到系统相册。
- App 不实现作品库。
- App 仅保存最近一次作品回流元信息。
- 设置页提供“清除最近作品回流”入口。
- 聊天页展示最近作品回流卡片。
- 角色回复中可引用最近一次召唤或作品主题。
- 清除最近作品回流时，UI 文案必须明确说明仅清除 App 内记录，不会删除系统相册图片。

### 8.2 阶段产出

- 用户可以保存截图。
- 保存后回到聊天，角色可以基于最近作品做文本反馈。
- App 不误导用户以为可以删除系统相册图片。

### 8.3 验收标准

- 截图保存成功耗时满足 PRD 阈值。
- 低存储、保存失败、权限异常时有明确提示。
- 清除最近作品回流后，聊天不再引用该作品上下文。
- 清除最近作品回流的确认文案不误导用户认为 App 会删除系统相册文件。

## 9. 阶段 6：设置、安全与数据治理

目标：补齐 Alpha 前必须具备的治理能力。

### 9.1 主要任务

- 设置页展示隐私说明、ARCore 隐私告知、权限说明。
- 提供账号数据删除入口。
- 提供聊天记录/记忆清理入口。
- 提供最近作品回流清除入口。
- 实现未成年人自述后的受限规则。
- 实现危机内容安全模式。
- 明确角色不能宣称自己是真人、有意识或能替代现实关系。
- 为未成年人自述和危机内容准备可回归测试的固定输入样例。
- 前端和后端需要共同支持安全模式状态：后端收紧回复，前端避免展示主动召唤邀约类文案。

### 9.2 阶段产出

- 用户可查看数据使用说明。
- 用户可清理关键数据。
- 安全策略可通过测试用例验证。

### 9.3 验收标准

- 未成年人自述后，暧昧表达、深度陪伴表达、主动召唤邀请被收紧。
- 危机内容触发固定安全回复和求助引导。
- 数据删除或清理失败时有错误提示和重试。
- 输入明确自述未成年的测试句后，系统返回受限模式回复，且不展示主动召唤邀约文案。
- 输入危机内容测试句后，系统返回固定安全文案，并在短时会话内保持更保守的回复风格。

## 10. 阶段 7：Alpha 测试准备

目标：形成可给 20-50 人测试的版本。

### 10.1 主要任务

- 接入核心埋点。
- 按 PRD 指标拆分聊天漏斗、AR 漏斗、fallback 漏斗。
- 准备 Alpha 测试设备矩阵。
- 准备测试账号、测试锚点图、测试角色资产。
- 输出已知问题列表。
- 固化 Alpha 测试反馈表。

### 10.2 核心埋点

- 登录完成率。
- 创角完成率。
- 首次聊天完成率。
- 聊天首条回复时间。
- 进入召唤页率。
- AR 初始化成功率。
- 图像锚定识别成功率。
- 角色放置完成率。
- 截图保存率。
- 拍摄后聊天回流率。
- fallback 使用率。
- 召唤页 crash-free rate。

### 10.3 验收标准

- Alpha 包可安装、可登录、可聊天、可召唤、可截图、可回流聊天。
- 已知高优先级崩溃清零。
- 所有 PRD P0/P1 场景均有测试记录。

## 11. 推荐开发顺序

1. 工程基线与导航。
2. mock 登录、隐私同意、创角、聊天首页。
3. mock LLM 正常/安全 preset 与安全策略占位。
4. 角色 GLB 接入与本地展示。
5. 召唤入口、锚点图展示。
6. fallback 召唤页。
7. ARCore 图像锚定。
8. 截图保存与聊天回流。
9. 设置、隐私、删除、治理。
10. 埋点、性能、Alpha 测试。

## 12. 关键阻塞项

- 首版角色资产 `E:\aic\feiyingfbx\未命名.fbx` 是否稳定。
- `FBX -> GLB` 转换质量是否可接受。
- 官方锚点图能否通过 ARCore 图像质量验证。
- LLM 服务是否可稳定返回，首条回复是否能控制在 `<= 4 秒`。
- Android 真机是否具备足够测试覆盖，尤其是 AR 支持设备和非 AR 设备。

## 13. 默认假设

- 当前 PRD v0.4 作为产品需求基线。
- 当前技术设计文档 v0.1 作为技术实现基线。
- 首版只支持 1 个用户、1 个角色、1 张官方锚点图。
- 首版不做作品库、不做录像、不做房间扫描、不做空间记忆。
- 首版后端可以先用 mock 或轻量服务替代，但接口边界必须按正式服务设计。
