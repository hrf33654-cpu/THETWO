# 开发流程文档：二次元聊天陪伴 + AR 召唤 App MVP

## 文档信息

- 文档版本：`v0.6`
- 文档状态：`In Progress`
- 更新时间：`2026-05-09`
- 关联 PRD：`PRD-二次元空间陪伴App-MVP.md`

- 客户端平台：`Android 原生`

## 1. 目标与原则

本文档从当前工程现状出发，指导 THETWO 从“可演示的 MVP 真后端 + 真 LLM 联调版”推进到“可进行 Alpha 测试的版本”。它不再按初始设想机械推进阶段，而是按当前最能提升产品可用性的顺序安排开发。

当前开发主线调整为：

`真实聊天能力 -> 会话持久化 -> Alpha 埋点与稳定性 -> 3D 角色资产 -> EasyAR 锚点召唤 -> EasyAR 平面稳定 -> 发布生产化`

开发原则：

- 聊天是主闭环，AR 是增强模块；不能为了 AR 推迟真实聊天能力。
- 先把已经跑通的 MVP 变成稳定、可回归、可测试版本，再扩大能力边界。
- fallback 召唤已经具备 MVP 价值，真 AR 应作为后续增强，不阻塞 Alpha。
- 首版只做最近一次截图回流，不做作品库、不做录像、不做房间理解。
- 所有登录、聊天、召唤、截图、删除和安全逻辑都必须能在真机上完整验证。
- 每一阶段结束必须有可演示版本、明确验收标准和失败回退路径。

## 2. 当前工程状态

当前工程已经完成：

- 工程基线：`com.thetwo.app`、单 Activity、Compose Navigation、基础 ViewModel/Repository 分层。
- 登录与创角：邮箱验证码 UI、隐私同意、开发验证码、远端创角保存。
- 真后端 Debug 联调：Android 客户端已默认通过 `AppContainer` 使用远端 Repository。
- 后端第一阶段：`Node + TypeScript + Express + SQLite`，已提供登录、用户、聊天、作品回流和删除接口。
- 聊天主闭环：远端历史拉取、远端消息发送、本地发送状态、失败重试、受限模式展示。
- 真实 LLM 公网联调：服务器 `.env` 已配置模型环境变量，`/chat/send` 已通过全新账号与 Android 实机验证。
- P1 第一阶段：`Preferences DataStore` 会话持久化、`Launch` 冷启动恢复、统一 401 清理、最小埋点骨架已接入代码。
- fallback 召唤闭环：CameraX 相机预览、纯屏 fallback、角色卡片拖动/缩放/旋转。
- 截图与回流：截图保存、`PUT /me/recent-capture`、回到聊天页展示最近作品。
- 设置与治理入口：清除最近作品、清空聊天、删除账号入口已调用后端接口。

当前仍未完成：

- 流式输出、会话摘要和长期记忆。
- Alpha 真机设备矩阵、系统化手测记录和已知问题清单。
- `FBX -> GLB -> 客户端加载` 的真实角色资产链路。
- EasyAR 初始化、锚点图识别、平面稳定和丢锚恢复。
- Release 可用的正式域名、HTTPS、真实邮件验证码、生产日志、备份和监控。

当前状态定义：

`聊天主闭环 + fallback 召唤闭环 + 真后端 + 真 LLM Debug 联调版`，不是 `PRD 完整目标版`。

## 3. 下一阶段总路线

后续开发按 P0 到 P3 推进。P0 和 P1 是 Alpha 前必须完成的主线；P2 是增强体验；P3 是正式发布生产化。

推荐顺序：

1. 补齐安全模式强化、超时重试细节、会话摘要和基础记忆。
2. 用真机完成 Alpha 核心链路回归，并整理设备矩阵与已知问题清单。
3. 跑通 `character.glb` 客户端加载和失败兜底。
4. 接入 EasyAR Sense 图像跟踪与平面识别，保持 fallback 不受影响。
5. 做域名、HTTPS、真实邮件、数据库迁移和生产运维。

## 4. P0：真实聊天与安全闭环

目标：把当前“代码已接入真实 LLM”的聊天链路推进到“线上已配置、真机已联调、可失败恢复、可安全收敛”的核心产品能力。

主要任务：

- 后端接入真实 LLM 服务，优先复用 `LLM_BASE_URL`、`LLM_API_KEY` 配置入口。
- 将 `POST /chat/send` 的 mock 回复替换为服务端统一封装的模型调用。
- 设计最小 prompt 结构：角色身份、关系边界、安全规则、最近会话上下文、最近作品回流。
- 增加 LLM 超时、失败、空回复和上游错误处理，客户端继续使用现有失败重试入口。
- 保持安全模式由后端返回 `NORMAL / RESTRICTED`，前端只消费结果并调整 UI 文案。
- 增加会话摘要与基础记忆的服务端占位，先服务于最近对话和最近作品，不做复杂长期记忆系统。
- 准备未成年人自述、危机内容、越界关系表达的固定测试输入。

阶段产出：

- 用户发送消息后，后端返回真实 LLM 回复。
- 回复风格能使用角色资料和最近作品上下文。
- 网络失败、LLM 超时、LLM 错误时，客户端有明确失败状态和重试入口。
- 安全/受限模式继续由后端决定，并能被客户端正确展示。

验收标准：

- 真实 LLM 可返回有效回复，不再依赖 `backend/src/reply.ts` 的固定 mock 文案作为主路径。
- 首条回复目标时间 `<= 4 秒`；超过目标时有明确等待、失败或重试策略。
- 失败消息可重试，不会造成重复插入不可区分的脏数据。
- 输入明确未成年人或危机内容测试句后，后端返回 `RESTRICTED`，前端不展示主动召唤邀约类文案。
- 至少有单测或服务端接口测试覆盖正常回复、受限回复、上游失败和空输入。
- 至少有一轮真机手测记录覆盖登录、创角、聊天、失败重试和受限模式。

当前状态注记：

- P0 的“真实 LLM 环境配置、公网联调、旧占位回复替换”已完成。
- P0 当前剩余重点转为安全强化、失败回退质量和摘要/记忆补齐。
- P1 的“会话持久化、冷启动恢复、统一 401 清理、最小埋点骨架”已完成代码接入。
- P1 当前剩余重点转为 Alpha 真机回归、设备矩阵和手测清单执行。

## 5. P1：会话持久化、埋点与 Alpha 稳定性

目标：让当前 Debug MVP 能被 20-50 人连续测试，而不是只能单次演示。

主要任务：

- 客户端持久化 `sessionToken`、用户邮箱、角色资料、最近作品回流和 AR/相机隐私确认状态。
- App 冷启动时优先恢复本地会话，再通过 `GET /me` 校验远端登录态。
- 401 或会话失效时清理本地认证状态，并明确回到登录页。
- 增加最小埋点骨架，先支持 Debug 本地日志，后续再替换正式分析平台。
- 埋点覆盖登录完成、创角完成、首次聊天完成、进入召唤页、截图保存、聊天回流、清理数据和账号删除。
- 建立 Alpha 真机设备矩阵，至少区分支持 AR 与不支持 AR 的设备。
- 输出 Alpha 手测清单和已知问题列表。

阶段产出：

- 重启 App 后能恢复登录态和关键上下文。
- 核心漏斗至少有最小埋点，不再只能靠人工描述判断完成率。
- Alpha 手测清单文档已补齐，Debug 包可交给小范围用户安装测试。

验收标准：

- 重启后保留登录态；远端会话有效时直接恢复到聊天页或必要的创角页。
- 远端会话失效时，客户端能清理本地状态并回到登录页。
- 核心漏斗有最小埋点：登录、创角、发消息、进入召唤、截图保存、回流聊天、删除数据。
- Debug 包可在真机完整走通：登录、聊天、召唤 fallback、截图、回流、清除最近作品、清空聊天、删除账号。
- 至少完成一轮支持 AR 设备和非 AR 设备的手测记录。
- 已知高优先级崩溃清零，阻断级问题必须进入已知问题列表。

当前状态注记：

- DataStore 会话持久化、Launch 启动恢复、统一 401 清理和 Debug 埋点日志已经完成。
- 当前未完成的是“按清单做真机回归并沉淀 Alpha 结果”。

## 6. P2：3D 角色与 EasyAR 召唤主路径

目标：在不破坏 fallback 闭环的前提下，把召唤体验从 2D 占位推进到 3D 角色、EasyAR 锚点图召唤和平面稳定。

主要任务：

- 完成首版 3D 角色资产交付，将个人本机源文件路径降级为来源备注。
- 资产交付以仓库可访问路径或明确制品包为准，不能依赖 `E:\aic\...` 这类个人路径。
- 跑通 `FBX -> GLB` 离线转换链路，输出 `character.glb` 和资产说明。
- 客户端加载 `character.glb`，提供加载中、加载失败、重试和 fallback 兜底。
- 接入 EasyAR Sense SDK 初始化和 AR Optional 检测。
- 接入 EasyAR Image Tracking / ImageTarget，识别官方锚点图后显示角色。
- 在锚点图召唤成功后，若设备支持平面识别，则启用平面稳定，让角色稳定放到检测到的真实平面上。
- 支持角色相对位移、旋转、缩放、重置。
- 实现短时丢锚保持，超过阈值提示重新对准。
- 保持 fallback 模式可用；EasyAR 初始化失败、锚点图识别失败、平面识别不可用、权限拒绝、设备能力不足时都不能阻断召唤闭环。

阶段产出：

- `character.glb` 可在客户端稳定加载。
- 支持 EasyAR 图像跟踪的设备能识别官方锚点图并显示角色。
- 支持平面识别的设备能在锚点召唤成功后启用平面稳定。
- 不支持 EasyAR 或 EasyAR 初始化失败时，用户仍能使用 fallback 召唤和截图。

验收标准：

- `character.glb` 可加载，有明确失败页和重试入口。
- 官方锚点图资源来自项目仓库或明确制品包，不依赖用户下载目录或个人机器路径。
- EasyAR 图像跟踪可用设备能识别官方锚点图并显示角色。
- 平面识别不可用时，保持锚点图平面召唤，不阻断截图。
- 不支持 EasyAR 的设备稳定进入 fallback，不崩溃、不阻断截图保存。
- EasyAR、平面稳定与 fallback 漏斗数据能区分统计。
- 召唤页核心 crash-free 指标达到 PRD 要求或进入 Alpha 风险清单。

## 7. P3：发布生产化

目标：将 Debug 联调能力整理成可发布、可运维、可追踪的问题闭环。

主要任务：

- 配置正式域名和 HTTPS，Release 不再使用明文 IP。
- 正式环境禁用开发验证码提示，接入真实邮件验证码服务。
- 梳理 Debug、Staging、Release 环境配置，避免生产包误连开发接口。
- 评估从 SQLite 迁移到 PostgreSQL 的时机，迁移前先补备份和回滚方案。
- 增加后端结构化日志、错误日志和基础健康检查。
- 建立数据库备份策略和最小监控告警。
- 清理仓库中的本地构建产物、无关 IDE 状态和不应提交的环境文件。

阶段产出：

- Release 包有可用的 HTTPS API。
- 生产配置不暴露开发验证码和本地调试入口。
- 后端有基础运维能力，问题能定位、数据能恢复。

验收标准：

- 域名 HTTPS 可用，Release 不走明文 IP。
- 生产配置不暴露开发验证码。
- 后端有基础日志与备份策略。
- 健康检查可用于判断服务是否在线。
- 关键环境变量和密钥不进入仓库。

## 8. 暂不做事项

为控制 MVP 范围，以下事项当前不进入下一阶段主线：

- 不先扩作品库；只保留最近一次作品回流。
- 不先做录像、房间扫描、空间理解或空间记忆。
- 不先做多角色、多资产上传或复杂角色编辑器。
- 不先做完整社区、分享、推荐或作品管理系统。
- 不先追求复杂 AR 表现；EasyAR 锚点召唤跑通前，fallback 是可交付主路径。
- 不先投入完整生产运维体系；P3 前只补最低限度的可观测和备份准备。

## 9. 阻塞项与风险

当前关键阻塞项：

- LLM 服务供应商、模型、费用、延迟和可用性尚未最终确认。
- 安全模式仍是关键词 + mock 规则，真实多轮安全策略未完成。
- `sessionToken` 仍以内存态为主，重启后需要重新登录。
- 角色资产尚未形成稳定的 `GLB` 运行时产物。
- 官方锚点图尚未完成 EasyAR 图像跟踪质量验证。
- 域名、HTTPS、真实邮件验证码和 PostgreSQL 迁移尚未进入正式实现。

风险处理原则：

- 如果 LLM 接入不稳定，先保证聊天失败状态和重试闭环，不让用户误以为消息已成功。
- 如果 GLB 质量不稳定，继续以 fallback 召唤作为 Alpha 可测能力。
- 如果 EasyAR 设备覆盖不足，Alpha 设备矩阵必须明确标记 EasyAR、平面稳定与 fallback 分组。
- 如果生产化资源不足，Release 前至少完成 HTTPS、生产配置隔离和密钥治理。

## 10. Alpha 验收清单

Alpha 前至少完成以下验收：

- 登录：可请求验证码、可验证、隐私同意未勾选时不能提交。
- 创角：可保存角色资料，重新进入后能恢复。
- 聊天：真实 LLM 可回复，失败可重试，受限模式可触发。
- 会话：重启 App 后能恢复登录态；失效后能回到登录页。
- 召唤 fallback：相机预览和纯屏模式都可用。
- 截图：可保存到系统相册或应用图片目录。
- 回流：截图成功后聊天页能展示最近作品引用。
- 设置：清除最近作品、清空聊天、删除账号都有结果反馈。
- 埋点：核心漏斗事件至少能被记录。
- 真机：支持 AR 设备和非 AR 设备都完成一轮主链路测试。

## 11. 默认假设

- 当前 PRD v0.4 作为产品需求基线。
- 当前实现状态文档 v0.4 作为工程事实基线。
- 首版只支持 1 个用户、1 个角色、1 张官方锚点图。
- 后端接口边界继续按正式服务设计，客户端不直连模型服务。
- fallback 召唤闭环是 Alpha 可交付能力，真 AR 是后续增强能力。
