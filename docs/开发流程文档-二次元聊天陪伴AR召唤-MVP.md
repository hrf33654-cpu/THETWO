# 开发流程文档：二次元聊天陪伴 + AR 召唤 App MVP

## 1. 目标与原则

本文档从当前工程现状出发，指导 THETWO 从“可演示 MVP”推进到“可稳定 Alpha 测试，再逐步扩展 3D / EasyAR / 发布能力”的状态。

当前开发原则：

- 聊天是主闭环，AR 是增强模块
- fallback 召唤已经具备 MVP 价值，不因为 EasyAR 阻塞整个项目
- 当前阶段先确保：
  - 主链路稳定
  - 问题可追踪
  - 外部阻塞项和代码内问题分清

## 2. 当前工程阶段

当前工程已完成：

- `P0`：真实聊天与安全闭环已基本完成
- `P1`：会话持久化、401 清理、最小埋点、Alpha 基线已基本完成

当前工程进行中：

- `P2-A`：3D 运行时验证
- `P2-B`：EasyAR 主路径

当前工程未完成：

- `P3`：发布生产化

## 3. 当前开发主线

当前推荐主线调整为：

`聊天稳定性 -> Alpha 多设备沉淀 -> 3D 运行时验证 -> EasyAR 主路径 -> 发布生产化`

也就是说，EasyAR 当前不再被视为“纯代码内可闭环项”，而是一个**受外部 license 阻塞的增强主线**。

## 4. 当前阶段拆分

### P0：真实聊天与安全闭环

当前状态：`Done / 可用`

已完成：

- 真后端接入
- 真 LLM 接入
- 服务端摘要 / 记忆 / 安全状态
- 失败重试与基础安全模式

当前剩余：

- 继续排查上游模型服务额度 / 鉴权 / 套餐匹配问题
- 补更稳定的错误提示和回归

### P1：会话持久化与 Alpha 稳定化

当前状态：`Done / 可测`

已完成：

- DataStore 会话持久化
- Launch 冷启动恢复
- 统一 401 清理
- Debug 埋点骨架
- 一轮主链路真机回归

当前剩余：

- 补齐更多设备回归记录
- 继续复验已修问题

### P2-A：3D 运行时验证

当前状态：`In Progress`

已完成：

- Filament 运行时接入
- `placeholder_cube.glb` 占位立方体接入
- 3D 资源预检与失败回退
- `character.glb` 不兼容原因确认：
  - `493 bones`
  - Filament 当前手机预览链上限 `256`

当前剩余：

- 继续验证占位立方体 3D 链路
- 并行准备最终角色资产移动端版本

### P2-B：EasyAR 主路径

当前状态：`Blocked`

已完成：

- `EasyAR.aar` 已接入
- Android 包名与最新 key 已真实打包验证
- 召唤页已接入 EasyAR 初始化与 tracking 相关代码

当前阻塞：

- EasyAR 主路径被 **license `Invalid Key`** 阻塞
- 当前结论不是“设备不支持”，而是“license 校验未通过”

因此当前验收结论：

- 真 AR 合屏不可验收
- 召唤页应以 fallback 为主能力

### P3：发布生产化

当前状态：`Not Started`

后续再做：

- Release 域名 / HTTPS
- 真实邮件验证码
- PostgreSQL 迁移
- 生产日志、监控、备份

## 5. 当前下一步优先级

当前建议顺序：

1. 继续等待 / 排查 EasyAR 官方 license 问题
2. 不被 EasyAR 卡住的前提下，继续推进聊天稳定性和召唤 fallback 体验
3. 并行准备最终角色资产的移动端版本
4. EasyAR 阻塞解除后，再继续真相机 + 锚点 + 立方体叠加

## 6. 当前不应再当作主线推进的事项

当前阶段不建议继续投入主时间在这些方向：

- 把 EasyAR 继续当作“纯代码问题”硬修
- 直接把 `character.glb` 强行接入真机 AR 主路径
- 先做全屏 AR UI 改造
- 先做真实 AR 截图合成

原因：

- EasyAR 当前主阻塞不是界面，而是 license
- 最终角色资产当前也不是可运行态

## 7. 当前结论口径

当前流程文档里的统一表述应为：

**项目当前主链路可用，P0/P1 已基本完成；P2 已进入 3D 与 EasyAR 阶段，但 EasyAR 主路径当前被 license `Invalid Key` 阻塞，因此现阶段应继续依赖 fallback 召唤闭环推进 Alpha。**
