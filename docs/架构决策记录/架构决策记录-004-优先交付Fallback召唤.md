# 架构决策记录 004：优先交付 EasyAR 锚点召唤

## 状态

`Accepted`

## 背景

EasyAR 平面识别依赖设备传感器、相机质量、光照、锚点图质量和 3D 资产稳定性。MVP 阶段如果一开始就把平面识别作为主入口，容易推迟可测试版本。

当前工程已经实现 CameraX fallback、纯屏 fallback、角色卡片拖动缩放旋转、截图保存和聊天回流。这条路径已经能验证“召唤 -> 截图 -> 回到聊天”的产品闭环。

## 决策

优先交付 EasyAR 锚点图召唤闭环，并将 CameraX 相机叠加和纯屏召唤作为降级路径。

平面识别不是独立主入口，而是在锚点召唤成功后用于把角色稳定放置到检测到的真实平面上。fallback 不是废弃方案，也不是临时调试页；它是 MVP 阶段正式支持的降级体验。

## 后果

收益：

- 锚点图召唤的演示稳定性高于直接平面识别。
- 截图回流和聊天上下文可以提前验证。
- 平面识别能力不足不会阻断 Alpha。

代价：

- 需要先准备高质量官方锚点图。
- 平面识别体验会后于锚点召唤完善。

## 当前适用范围

- 适用于 MVP 到 Alpha 阶段召唤功能。
- 如果后续 AR 交互方案变化，应先保留 EasyAR 锚点召唤和 fallback，再通过新 ADR 调整主路径。
