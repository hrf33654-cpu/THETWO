# 架构决策记录 008：摘要与基础记忆通过 LLM 异步生成

## 状态

`Accepted`

## 背景

THETWO 当前已经具备真实聊天主链路，但如果只依赖最近几条原始消息作为上下文，角色会很快“忘事”。  
同时，如果每轮聊天都同步做“回复 + 摘要 + 记忆”三次模型调用，会明显拉高首条回复延迟，并破坏聊天主闭环体验。

我们需要一种方案，让角色能保留稳定偏好和持续话题，又不把客户端、接口和延迟一起做复杂。

## 决策

摘要与基础记忆都由服务端调用现有 LLM 异步生成。

固定策略如下：

- 聊天主回复仍是同步主路径。
- 会话摘要与基础记忆不由客户端生成，不由客户端存储，也不做用户可见记忆面板。
- 只有当聊天历史总量已超过最近上下文窗口，且距离上次摘要后的新增消息数 `>= 8` 时，才触发摘要刷新。
- 摘要刷新完成后，再基于最新摘要与最近对话提取轻量记忆。
- 摘要和记忆都只作为后续 prompt 的服务端内部上下文。

当前落库模型：

- `chat_summaries`
- `memory_states`
- `safety_states`

## 后果

收益：

- 角色能记住稳定偏好、近期关系进展和最近作品主题。
- 不需要改 Android 聊天协议。
- 不需要在客户端引入记忆 UI 或复杂本地同步。
- 主回复延迟优先得到保障，摘要与记忆刷新失败不会阻断聊天。

代价：

- 摘要和记忆不是实时同步完成，可能会滞后一轮或几秒。
- 服务端模型调用次数增加。
- 服务端需要处理“主回复成功、异步总结失败”的日志与排查。

## 删除语义

- `DELETE /chat/history` 必须同时清空：
  - `chat_messages`
  - `chat_summaries`
  - `memory_states`
  - `safety_states`
- `DELETE /me` 继续删除用户所有关联数据。

## 当前适用范围

- 适用于聊天主闭环的上下文压缩、轻量长期记忆和短时安全状态。
- 不适用于用户可见记忆列表、记忆编辑功能或复杂结构化长期记忆系统。
